# Agent configuration architecture

This directory is the canonical source for AI-agent assets shared by Codex and Claude Code.

## Single source of truth

- `agent-config.yaml` is the only hand-edited source for shared agent settings.
- `skills/` is the canonical cross-agent skill tree and is exposed as `~/.agents/skills`.
- Claude Code receives generated symlinks under `~/.claude/skills` so it sees the same skill set.
- Codex receives the same skill tree through the shared `~/.agents/skills` location and the local workflow plugin adapter.
- Agent-native files are generated by `../../scripts/generate-agent-configs.py`.
- Codex sandbox writable roots for shared `agmsg` state (`~/.agents/skills/agmsg/db`, `~/.agents/skills/agmsg/teams`, and `~/.agents/skills/agmsg/run`) are specified under `codex.sandbox_workspace_write` in `agent-config.yaml`.
- Directly invoked `agmsg` scripts must use chezmoi `executable_` source prefixes so `chezmoi apply` preserves execute bits.
- Shared long-term memory is represented as the optional `cognee_memory` MCP server. Run Cognee as a long-lived HTTP MCP service and let both agents connect to that same service.

Generated files include:

- `home/.chezmoitemplates/codex-config-managed.toml`
- `home/dot_codex/modify_private_config.toml`
- `home/dot_codex/<profile>.config.toml` (one per `model_profiles` entry)
- `home/.chezmoitemplates/claude-settings-managed.json`
- `home/dot_claude/private_mcp.json.tmpl`
- `home/dot_claude/agents/express-explorer.md`
- `home/dot_claude/skills/**/symlink_*.tmpl`
- `home/dot_agents/model-profiles.env`
- `home/dot_agents/plugins/marketplace.json`
- `home/dot_agents/plugins/mryfmo-dev-workflows/.codex-plugin/plugin.json`

## Parity policy

1. Every MCP server in `agent-config.yaml` must target both agents: Codex and Claude Code.
2. Every generated agent-native MCP config must contain the same server names.
3. MCP servers remain disabled by default. Enable them locally only after confirming credentials, filesystem scope, and network behavior.
4. GitHub MCP must use the maintained official GitHub server (`ghcr.io/github/github-mcp-server`), not deprecated `@modelcontextprotocol/server-github`.
5. Public dotfiles must not contain token literals. Credential-bearing servers inherit environment variables or refer to private configuration only.
6. Claude plugins are not enabled by `settings.json` unless this repository also installs the marketplace/plugin. Shared workflows should live in skills first.
7. Do not hand-edit generated files unless you immediately move the change back into `agent-config.yaml` and regenerate.
8. Cognee memory stays disabled in public defaults. Enable it locally only after starting Cognee MCP and providing LLM/API/DB secrets from private environment or private chezmoi state.
9. For implementation tasks shared between Codex and Claude Code, use separate worktrees or make one agent a reviewer; do not let both write to the same worktree. This is an operational guideline and is intentionally not enforced by `validate-agent-assets.py`.

## Codex runtime state

Codex writes stable runtime state into `~/.codex/config.toml`, so the file is managed through `home/dot_codex/modify_private_config.toml` instead of a full-file template. The generated managed baseline lives at `home/.chezmoitemplates/codex-config-managed.toml`.

The merge script keeps managed settings from `agent-config.yaml` authoritative while preserving these Codex-owned table prefixes from the live target when they exist:

- `hooks.state`
- `marketplaces`
- `tui.model_availability_nux`
- `projects`

Current-target tables outside the generated baseline are also preserved so local Codex additions are not lost during `chezmoi apply`.

## Shared Cognee memory

`cognee_memory` is the cross-agent memory access layer. It is intentionally an MCP server rather than a Claude-only memory plugin so that both agents can use the same backend:

```text
Codex ────────┐
Claude Code ──┴── Cognee MCP HTTP server ── shared memory backend
```

Recommended local service shape:

- Cognee MCP listens at `http://127.0.0.1:8000/mcp`.
- Enable installation with chezmoi data such as `cognee.install: true`; the default remains `false`.
- The opt-in installer runs `install/common/cognee.sh`, which installs `cognee-mcp` with `uv tool install`.
- Start the local HTTP MCP service with `start-cognee-mcp` after applying chezmoi.
- Secrets such as `LLM_API_KEY`, database URLs, and cloud tokens live outside this public repository.
- Cognee deletion tools such as `prune`, `delete`, `delete_dataset`, and `forget_memory` are not part of the default safe tool set for generated agent configs.
- Claude Code also receives the server definition disabled by default; if enabling it manually, keep destructive memory tools restricted by local policy.

Local setup example:

```bash
# Set data.cognee.install: true in ~/.config/chezmoi/chezmoi.yaml,
# then apply the managed run_once installer.
chezmoi apply
start-cognee-mcp
```

The `start-cognee-mcp` helper starts Cognee with the endpoint expected by generated agent configs. Override `COGNEE_MCP_HOST`, `COGNEE_MCP_PORT`, `COGNEE_MCP_PATH`, or `COGNEE_MCP_TRANSPORT` only if you also update the agent MCP URL accordingly.

Use Cognee for durable facts only:

- user preferences that will remain useful
- repository conventions and architectural decisions
- recurring setup or CI pitfalls
- fixes that future agents should remember

Do not store secrets, raw logs, temporary PR state, one-off TODO lists, or noisy command output.

## Commands

Generate agent-native files:

```bash
uv run --with pyyaml scripts/generate-agent-configs.py
```

Verify generated files are up to date:

```bash
uv run --with pyyaml scripts/generate-agent-configs.py --check
```

Run the full agent asset validator:

```bash
uv run --with pyyaml scripts/validate-agent-assets.py
```

After applying chezmoi, prove the active HOME runtime files match this source tree:

```bash
scripts/check-agent-runtime.py
```

The validator checks generated-file freshness, skill parity, MCP parity, disabled-by-default safety, deprecated MCP packages, schema/parse validity, and obvious committed secrets. The runtime doctor checks the applied Codex, Claude Code, skill, and hook files in `$HOME` without modifying them.
