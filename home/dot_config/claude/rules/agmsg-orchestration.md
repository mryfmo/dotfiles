## agmsg orchestration

- When the operator requests agmsg/Codex collaboration or an orchestration regime is active, act only as orchestrator: lightweight grep/read, judgment, task authoring, and acceptance review.
- Delegate all repository-mutating work — file edits, builds, test runs, git state changes — to resident Codex workers in herdr panes, at most one resident worker per git worktree; within a single worktree assign tasks sequentially. Parallelism is achieved only by adding worktrees, never through parallel codex exec or per-task Codex spawning. agmsg/herdr control-plane commands (delivery.sh, watch.sh, actas-claim.sh, send.sh, join.sh, herdr agent/pane) are orchestrator-side and exempt.
- A parallel assignment is valid only when every concurrent worker has all four of: (1) its own git worktree, registered as its agmsg `project` path; (2) the shared default agmsg store — never assign per-worker `AGMSG_STORAGE_PATH` stores under the same-repo parallel regime: delivery is identity-addressed (`to_agent`) and `whoami` resolves per worktree path, so inboxes are already isolated, and the single activation watcher then sees every RESULT and PONG without extra watchers (which `watch.sh` actas locking cannot support for one claimed identity anyway); (3) an `-aNNN` instance suffix on its identity — under a parallel assignment ALL concurrent workers carry a suffix, including the first; (4) an AGMSG-TASK whose `allowed_files`, after expanding any task-file sections, is pairwise disjoint from every other in-flight task. The orchestrator verifies disjointness at assignment time and performs all cross-worktree integration (merge, rebase, conflict resolution) itself.
- At parallel-worker teardown run `delivery.sh set off <type> <worker worktree path>`; teardown kills every watcher on that exact path regardless of type.
- Review every RESULT adversarially across correctness, regressions, security, and reporting omissions: try to refute it, independently re-derive findings, and never treat sampled spot-checks as full verification.
- Do not idle-wait while worker work is in flight; prepare or delegate independent work.
- Acceptance review, adversarial RESULT review, and review-profile work remain orchestrator-side (Claude); never delegate review, acceptance, or `make require-crit-review` to a Codex worker, and keep `make require-crit-review` as the orchestrator's final integration step. Revisit this clause only if worker-side model capability surpasses the orchestrator tier.
- On regime activation, verify delivery with `delivery.sh status <type> <repo>`; if it is weaker than `both`, run `delivery.sh set both <type> <repo>`, start the SessionStart-printed `watch.sh <session_id> <repo> <type>` command as a persistent in-session monitor, and claim exclusivity with `actas-claim.sh <project> <type> <name> <session_id>`.
- Detect worker completion only when an AGMSG-RESULT message arrives through monitor/turn delivery; send worker liveness and status checks only as AGMSG-PING/AGMSG-PONG over the bus, never by reading worker panes or screens; limit pane interaction to prompt injection and the submit key, never infer completion from pane or agent status, and never use ad-hoc polling sleep loops.
- If an out-of-band Codex completion signal is ever needed, use the official `notify` config: the `agent-turn-complete` event sends a JSON payload to an external command.
- Give each physical agent one unique identity: `<runtime>-<profile>-<project-suffix>` (for example, `codex-standard-dot` for dotfiles or a `-flue` suffix for flue-pi). The `<project-suffix>` derives from the repository, not the checkout — every worktree of the same repo shares it. Model IDs never appear in identities — `model_profiles` in `agent-config.yaml` is the only place models are named. A solo resident worker needs no instance suffix; whenever workers run in parallel (sharing runtime, profile, and project-suffix), ALL of them append `-aNNN` (`-a001`, `-a002`, …).
- Before any join, search every `~/.agents/skills/agmsg/teams/*/config.json` for the candidate name; on collision, choose a suffixed unique name and never reuse one identity for different physical agents.
- Always register `project` as the worker's own real working-tree path (the dedicated worktree for parallel workers), byte-identical across join, delivery setup, and hook argv — identity resolution is exact string equality, so a trailing slash or unresolved symlink silently orphans the inbox; `$HOME` registrations are forbidden because they create codex-hook ambiguity and steal inbox messages.
- At worker setup run delivery.sh set turn codex <worker worktree path> so the worker Stop hook auto-delivers inbox messages each turn from the tree-scoped .codex/hooks.json (gitignored), and confirm the pane's codex environment has `AGMSG_STORAGE_PATH` unset — storage resolution is env-only, so an inherited stray value silently reroutes the worker to a different database; pane nudges are only generic wakes for idle workers — message content always flows over the bus.
- Reserve `AGMSG_STORAGE_PATH` store separation for concurrent regimes in different projects (for example flue-pi); when a separate store is used, the orchestrator must prefix the SAME AGMSG_STORAGE_PATH on its send/watch/history calls for that worker, or results and pongs land in an unreachable database. Same-repo parallel workers always share the default store.
- Accept tasks that change live desktop behavior (herdr layout/session, pane lifecycle, or delivery hooks) only after live end-to-end verification covers both a fresh session and a persisted-session restore; unit tests and static checks alone are insufficient.
- Invoke the `agmsg-orchestration` skill for AGMSG-TASK/RESULT/ACCEPTANCE message contracts and playbooks.
